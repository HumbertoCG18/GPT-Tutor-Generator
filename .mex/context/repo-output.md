# Repo Output Format

This documents the structure of the GitHub repository that the tool generates — the one used as a Claude Projects knowledge base.

## Critical Files

```
course/COURSE_MAP.md          # pedagogical map — primary entry point for the Claude tutor
course/FILE_MAP.md            # routing index with priority — used by the tutor to navigate files
course/SYLLABUS.md            # syllabus generated from SubjectProfile
course/GLOSSARY.md            # glossary seeded and refined from course content
student/STUDENT_STATE.md      # student profile and progress
exercises/EXERCISE_INDEX.md   # exercise index
build/claude-knowledge/bundle.seed.json   # internal seed — not loaded eagerly
setup/INSTRUCOES_CLAUDE_PROJETO.md        # tutor instructions for Claude
setup/INSTRUCOES_GPT_PROJETO.md           # tutor instructions for GPT
setup/INSTRUCOES_GEMINI_PROJETO.md        # tutor instructions for Gemini
```

## How the Tutor Uses These Files

1. On first session, the Claude tutor reads `COURSE_MAP.md` and `FILE_MAP.md`.
2. It performs the **First Session Protocol**: maps files to course units using `FILE_MAP.md` as a navigation index.
3. `STUDENT_STATE.md` tracks the student's progress across sessions.
4. Internal files (e.g., `bundle.seed.json`) live under `build/` to avoid eager loading by Claude Projects.

## FILE_MAP Format

`FILE_MAP.md` is a routing index generated during build. It contains:
- File paths with priority weights
- No pre-assigned categories (categories are assigned by the tutor during First Session Protocol)

The matching and routing logic lives in `builder/routing/file_map.py`.

## Multi-LLM Instruction Files

The build generates separate instruction files for each supported LLM from a single run:
- `INSTRUCOES_CLAUDE_PROJETO.md` — for Claude Projects
- `INSTRUCOES_GPT_PROJETO.md` — for GPT
- `INSTRUCOES_GEMINI_PROJETO.md` — for Gemini

Controlled by `preferred_llm` and `github_url` fields in `SubjectProfile` (`src/models/core.py`).

## What Does NOT Exist in the Generated Repo

- No LLM API calls happen at build time — the LLM is used only at runtime inside the generated repo.
- No auto-categorization — categorization is done by the tutor at runtime via First Session Protocol.
